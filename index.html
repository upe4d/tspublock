Когда пользователь подключается к MTProxy, ТСПУ вклинивается в соединение и отправляет TCP RST пакеты со своих IP — сервер думает что соединение разорвано и закрывает его. Помочь пополнить базу

Если у вас есть свой MTProxy сервер — запустите скрипт. Он поработает 30 секунд, соберёт IP ТСПУ которые атакуют именно ваш сервер и отправит их в нашу общую базу.

Требования: Linux, root, iptables, curl. Запускать на сервере с MTProxy.

curl -fsSL https://stats.gptru.pro:4443/rst/rst_submit.sh | sudo bash -s -- ВАШ_ПОРТ 30

Замените ВАШ_ПОРТ на порт вашего прокси (обычно 443).

Исходный код скрипта

Что происходит?

ТСПУ — «чёрные ящики» РКН на сетях операторов. При обнаружении MTProxy-подключения они шлют поддельные TCP RST пакеты на прокси-сервер.

Сервер получает RST и думает что клиент сам разорвал соединение — Telegram handshake timeout. Клиент видит что прокси не работает.

Решение — дропать RST пакеты от известных ТСПУ адресов через iptables + ipset. Соединение не рвётся, Telegram работает стабильнее.

Установка на свой сервер

Требования: Linux, root, ipset.

1
Установить ipset
apt-get install -y ipset
2
Скачать и применить блок-лист
Одна команда — скачивает актуальный список и настраивает iptables:
curl -fsSL "https://stats.gptru.pro:4443/rst/api.php?action=export&fmt=iptables" -o /tmp/tspublock.sh && bash /tmp/tspublock.sh
3
Проверить что работает
iptables -L TSPUBLOCK -v -n
Столбец pkts должен расти — это заблокированные RST пакеты.
4
Сохранить правила
ipset save > /etc/ipset.conf && iptables-save > /etc/iptables/rules.v4
5
Автообновление каждые 2 минуты
curl -fsSL https://stats.gptru.pro:4443/rst/rst_collect.sh -o /opt/rst_collect.sh && chmod +x /opt/rst_collect.sh && (crontab -l 2>/dev/null; echo "*/2 * * * * /opt/rst_collect.sh >> /var/log/rst_collect.log 2>&1") | crontab -

Для MikroTik дома

Скачайте .rsc файл и импортируйте в роутер:

/import file=tspuips_mikrotik.rsc

Затем добавьте правило фаервола:

/ip firewall filter add chain=input protocol=tcp tcp-flags=rst src-address-list=TSPUIPS action=drop
